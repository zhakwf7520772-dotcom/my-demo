# C++ 学习项目需求说明书 V2（可执行版）

## 1. 文档目的
本版本用于解决“知道要做什么，但不知道如何一步步做”的问题。

你可以把这份文档当成一个实施手册：
- 明确每一阶段要产出什么
- 明确每一阶段怎么做
- 明确每一阶段怎么验收

## 2. 项目定义

### 2.1 项目名称
`rpc-demo`：基于 C++ 的微服务学习项目（RPC + Redis + MySQL + Docker）

### 2.2 学习目标（可检验）
完成项目后，你应能独立完成：
1. 用 Protobuf + gRPC 定义并实现跨服务 RPC。
2. 用 MySQL 做结构化数据存储与查询。
3. 用 Redis 做缓存命中、失效与一致性处理。
4. 用 Docker Compose 一键启动完整开发环境。
5. 用日志和基础测试定位问题。

### 2.3 业务场景
实现“用户-任务”系统：
- 用户服务：创建用户、查询用户。
- 任务服务：创建任务、查询任务列表、更新任务状态。
- 任务服务创建任务时，必须 RPC 调用用户服务验证 `user_id` 是否存在。

## 3. 范围与边界

### 3.1 本期范围（必须做）
1. 两个 C++ gRPC 服务：`user_service`、`task_service`。
2. MySQL 两张表：`users`、`tasks`。
3. Redis 缓存：用户详情、用户任务列表。
4. Docker Compose：启动 mysql、redis、两个服务。
5. 基础日志、错误码、配置文件。

### 3.2 本期不做（避免过度设计）
1. 认证鉴权（JWT/OAuth）。
2. 服务注册发现。
3. 消息队列。
4. 高可用集群。

## 4. 目标架构

```text
client (grpcurl / test client)
        |
        v
  task_service ----------------------> user_service
       |                                   |
       |                                   |
       v                                   v
     MySQL                               MySQL
       ^                                   ^
       |                                   |
      Redis (task:list:{user_id})        Redis (user:{id})
```

说明：
- `task_service` 写任务前先调用 `user_service.GetUser`。
- 两个服务可共享同一 MySQL 实例，但使用独立 repository 层。
- Redis 只做缓存，不做真数据源。

## 5. 详细功能需求

### 5.1 用户服务（user_service）

#### 5.1.1 创建用户
- 输入：`name`、`email`
- 规则：
  1. `name` 非空，长度 1~64
  2. `email` 非空，格式合法
  3. `email` 全局唯一
- 输出：`user_id`、`created_at`
- 错误：
  - 参数错误
  - 邮箱已存在
  - DB 异常

#### 5.1.2 查询用户
- 输入：`user_id`
- 流程：
  1. 先查 Redis `user:{id}`
  2. miss 后查 MySQL
  3. 查到后回填 Redis（TTL）
- 输出：用户信息
- 错误：用户不存在

### 5.2 任务服务（task_service）

#### 5.2.1 创建任务
- 输入：`user_id`、`title`、`content`
- 规则：
  1. `title` 非空，长度 1~128
  2. 状态初始为 `todo`
  3. 创建前 RPC 校验用户存在
- 流程：
  1. RPC 调用用户服务
  2. 写 MySQL
  3. 删除 Redis 列表缓存 `task:list:{user_id}`
- 输出：`task_id`

#### 5.2.2 查询任务列表
- 输入：`user_id`、分页参数 `page`、`page_size`
- 流程：
  1. 先查 Redis 列表缓存
  2. miss 查 MySQL
  3. 回填 Redis（TTL）
- 输出：任务列表 + 分页信息

#### 5.2.3 更新任务状态
- 输入：`task_id`、`status`
- 规则：状态只能是 `todo/doing/done`
- 流程：更新 MySQL 后失效对应列表缓存

## 6. 数据库设计（MySQL）

### 6.1 users 表
```sql
CREATE TABLE IF NOT EXISTS users (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(64) NOT NULL,
  email VARCHAR(128) NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_email (email)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 6.2 tasks 表
```sql
CREATE TABLE IF NOT EXISTS tasks (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id BIGINT NOT NULL,
  title VARCHAR(128) NOT NULL,
  content TEXT,
  status VARCHAR(16) NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  KEY idx_user_id (user_id),
  KEY idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 7. RPC 接口设计（Proto）

### 7.1 user_service.proto（建议）
```proto
syntax = "proto3";
package user.v1;

service UserService {
  rpc CreateUser(CreateUserRequest) returns (CreateUserResponse);
  rpc GetUser(GetUserRequest) returns (GetUserResponse);
}

message CreateUserRequest {
  string name = 1;
  string email = 2;
}

message CreateUserResponse {
  int32 code = 1;
  string message = 2;
  int64 user_id = 3;
}

message GetUserRequest {
  int64 user_id = 1;
}

message User {
  int64 id = 1;
  string name = 2;
  string email = 3;
  string created_at = 4;
}

message GetUserResponse {
  int32 code = 1;
  string message = 2;
  User user = 3;
}
```

### 7.2 task_service.proto（建议）
```proto
syntax = "proto3";
package task.v1;

service TaskService {
  rpc CreateTask(CreateTaskRequest) returns (CreateTaskResponse);
  rpc ListTasks(ListTasksRequest) returns (ListTasksResponse);
  rpc UpdateTaskStatus(UpdateTaskStatusRequest) returns (UpdateTaskStatusResponse);
}

message Task {
  int64 id = 1;
  int64 user_id = 2;
  string title = 3;
  string content = 4;
  string status = 5;
  string created_at = 6;
}
```

## 8. 缓存设计（Redis）

### 8.1 Key 规范
- 用户详情：`user:{user_id}`
- 任务列表：`task:list:{user_id}:{page}:{page_size}`

### 8.2 TTL 建议
- 用户详情：300 秒
- 任务列表：120 秒

### 8.3 一致性策略
采用“先写库，后删缓存”：
1. 写操作成功后删除相关 key。
2. 读请求在 cache miss 时回源 DB 并重建缓存。

## 9. 错误码规范（建议）

| code | 含义 |
|---|---|
| 0 | 成功 |
| 40001 | 参数错误 |
| 40401 | 用户不存在 |
| 40402 | 任务不存在 |
| 40901 | 邮箱重复 |
| 50001 | 数据库错误 |
| 50002 | Redis 错误 |
| 50003 | RPC 下游错误 |

## 11. 分阶段实施步骤（一步一步做）

### 阶段 0：环境准备（0.5 天）
任务：
1. 安装 `docker`、`docker compose`、`cmake`、`g++`。
2. 准备 `docker-compose.yml`（mysql + redis）。
3. 启动并确认健康状态。

验收：
```bash
docker compose up -d
docker compose ps
```
看到 mysql/redis 为 `running` 且 `healthy`。

### 阶段 1：数据库初始化（0.5 天）
任务：
1. 创建 `sql/001_init.sql`，写入 `users/tasks` 建表语句。
2. 执行 SQL 初始化。
3. 用客户端检查表是否创建成功。

验收：
```bash
docker exec -it my-demo-mysql mysql -uroot -proot -e "USE demo; SHOW TABLES;"
```

### 阶段 2：Proto 定义与代码生成（0.5 天）
任务：
1. 编写 `proto/user_service.proto`、`proto/task_service.proto`。
2. 配置 CMake 的 protobuf/grpc 代码生成。
3. 保证服务端和客户端都能编译通过。

验收：
- 构建目录内有生成的 `*.pb.cc`、`*.grpc.pb.cc`。
- `cmake --build` 成功。

### 阶段 3：实现 user_service（1~1.5 天）
任务：
1. 实现 `CreateUser`（参数校验 + 唯一性 + 入库）。
2. 实现 `GetUser`（Redis 读缓存 + MySQL 回源 + 回填缓存）。
3. 增加日志和统一错误码。

验收：
1. 创建用户成功。
2. 同邮箱重复创建返回 `40901`。
3. 连续两次查询用户，第二次应缓存命中（看日志）。

### 阶段 4：实现 task_service（1~1.5 天）
任务：
1. 实现 `CreateTask`（调用 user_service 校验用户）。
2. 实现 `ListTasks`（缓存 + 分页）。
3. 实现 `UpdateTaskStatus`（更新后删缓存）。

验收：
1. 无效 `user_id` 创建任务失败（`50003` 或 `40401`）。
2. 创建任务后查询列表可见。
3. 更新状态后再次查询状态正确。

### 阶段 5：容器化服务（1 天）
任务：
1. 为两个服务分别编写 Dockerfile。
2. 在 compose 中加入 `user_service`、`task_service`。
3. 配置依赖、端口、环境变量。

验收：
```bash
docker compose up -d --build
docker compose ps
```
4 个服务全部正常。

### 阶段 6：联调与回归（1 天）
任务：
1. 准备 `scripts/smoke_test.sh`，按接口跑一遍核心流程。
2. 检查缓存命中、缓存失效、数据库结果一致性。
3. 整理 README 的启动/停止/清理命令。

验收：
- `smoke_test.sh` 全部通过。
- `docker compose down` + `up -d` 后可再次正常运行。

## 13. 质量门槛（最低要求）
1. 所有 RPC 接口都返回统一结构：`code/message/data`。
2. 关键日志必须含：`trace_id`（可先简单实现）、耗时、错误码。
3. 不允许把 SQL 写在 handler，必须经过 service/repository 分层。
4. 缓存错误不能影响主流程（降级到 DB）。

## 14. 风险与规避
1. 版本兼容风险：gRPC/Protobuf 版本不一致。
- 规避：固定版本，写进文档和 Dockerfile。

2. 缓存一致性风险：更新后读到旧数据。
- 规避：统一“写库成功后删缓存”策略。

3. 环境漂移风险：本地能跑，换机失败。
- 规避：全部依赖容器化，命令写入 README。
