# C++ 学习项目需求说明书（RPC + Redis + MySQL + Docker）

## 1. 项目背景
为了系统熟悉 `RPC`、`Redis`、`MySQL`、`Docker` 在实际工程中的协作方式，设计并实现一个可运行的 C++ 微服务示例项目。

项目定位：
- 以学习为主，业务复杂度低
- 工程要素完整（接口、缓存、持久化、容器化）
- 支持本地一键启动与联调

## 2. 项目目标
- 掌握 C++ 服务通过 RPC 调用的基本流程
- 掌握 Redis 在缓存与状态存储中的典型用法
- 掌握 MySQL 的表设计、CRUD 与事务基础
- 掌握 Docker / Docker Compose 的服务编排
- 形成可复用的项目骨架，便于后续扩展

## 3. 业务范围（建议）
采用“**用户与任务管理系统**”作为演示业务，包含两个服务：

1. `user_service`
- 用户注册
- 用户查询

2. `task_service`
- 创建任务
- 查询任务列表
- 任务状态更新

服务间关系：
- `task_service` 通过 RPC 调用 `user_service` 校验用户是否存在

## 4. 功能需求

### 4.1 用户模块
- 创建用户：`name`、`email`
- 按 `user_id` 查询用户
- 邮箱唯一性校验

### 4.2 任务模块
- 创建任务：`user_id`、`title`、`content`
- 查询某用户下任务列表
- 更新任务状态（`todo` / `doing` / `done`）

### 4.3 缓存需求（Redis）
- 用户查询结果缓存（key: `user:{id}`）
- 任务列表缓存（key: `task:list:{user_id}`）
- 写操作后进行缓存失效或更新

### 4.4 持久化需求（MySQL）
- `users`、`tasks` 两张核心表
- 任务表按 `user_id` 建索引
- 基础审计字段：`created_at`、`updated_at`

### 4.5 接口需求（RPC）
- 提供 proto 定义文件
- 提供请求/响应模型
- 统一错误码与错误信息

### 4.6 部署需求（Docker）
- 通过 `docker-compose up -d` 启动完整依赖
- 至少包含：`user_service`、`task_service`、`redis`、`mysql`
- 服务健康检查与启动顺序控制

## 5. 非功能需求
- 可观测性：基础日志（请求、错误、耗时）
- 配置化：端口、数据库地址、Redis 地址可通过配置文件或环境变量注入
- 可测试性：至少包含核心模块单元测试（建议）
- 可维护性：模块分层（接口层、业务层、存储层）

## 6. 里程碑建议

### M1：基础环境
- 搭建 C++ 工程与依赖管理
- 跑通 Redis/MySQL 容器

### M2：用户服务
- 完成 `user_service` RPC + MySQL + Redis

### M3：任务服务
- 完成 `task_service` RPC + MySQL + Redis
- 跑通服务间 RPC 调用

### M4：联调与容器化
- Dockerfile + docker-compose 完成
- 一键启动与基础验收

## 7. 验收标准
- 能通过 RPC 正常创建与查询用户
- 能通过 RPC 正常创建、查询、更新任务
- Redis 缓存命中与失效行为符合预期
- MySQL 中数据与业务操作一致
- 使用 Docker Compose 可在新环境 10 分钟内启动运行
